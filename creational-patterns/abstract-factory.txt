The Abstract Factory Pattern

1. Explanation
The Abstract Factory pattern is a creational design pattern that lets you produce families of related or dependent objects without specifying their concrete classes. While a Factory Method manages the creation of a single product type, an Abstract Factory acts as a "factory of factories," organizing multiple related factory methods into a unified suite.

Real-World Analogy: Think of a car manufacturing ecosystem. A brand like luxury-tier vs. economy-tier needs a complete family of matching components. The luxury factory line produces a luxury chassis, luxury leather seats, and a high-performance engine. The economy factory line manufactures a compact chassis, cloth seats, and a fuel-efficient low-displacement engine. You cannot mix a luxury chassis with an economy low-displacement engine without causing mechanical and structural failures. The Abstract Factory ensures that the whole family of components is completely uniform and structurally compatible.

2. Components Involved
- Abstract Factory (Interface): Declares a set of distinct factory methods for creating each of the abstract products that make up a product family (e.g., `DatabaseFactory`).
- Concrete Factory (Class): Implements the creation methods of the abstract factory, returning concrete products for a specific family variant (e.g., `PostgresFactory`, `OracleFactory`).
- Abstract Product (Interface): Defines blueprints for a set of distinct but related products in the system (e.g., `Connection`, `Command`).
- Concrete Product (Class): Implements the abstract product interfaces, grouped by their matching family variants.
- Client: Interacts exclusively with the Abstract Factory and Abstract Product interfaces, remaining entirely oblivious to concrete class instantiations.

3. Code Example: Cross-Database Compatibility Engine

// Abstract Product A
interface Connection {
    void establish();
}

// Abstract Product B
interface Command {
    void compileAndExecute(String query);
}

// Concrete Product Family 1: PostgreSQL Suite
class PostgresConnection implements Connection {
    public void establish() {
        System.out.println("Connecting securely to PostgreSQL via native TCP port 5432...");
    }
}

class PostgresCommand implements Command {
    public void compileAndExecute(String query) {
        System.out.println("Parsing PostgreSQL dialect syntax -> Executing: " + query);
    }
}

// Concrete Product Family 2: Oracle Suite
class OracleConnection implements Connection {
    public void establish() {
        System.out.println("Connecting to heavy Oracle DB cluster instance via port 1521...");
    }
}

class OracleCommand implements Command {
    public void compileAndExecute(String query) {
        System.out.println("Parsing Oracle PL/SQL dialect syntax -> Executing: " + query);
    }
}

// Abstract Factory Interface
interface DatabaseFactory {
    Connection createConnection();
    Command createCommand();
}

// Concrete Factory 1: Assembles PostgreSQL Environment Suite
class PostgresFactory implements DatabaseFactory {
    public Connection createConnection() { return new PostgresConnection(); }
    public Command createCommand() { return new PostgresCommand(); }
}

// Concrete Factory 2: Assembles Oracle Environment Suite
class OracleFactory implements DatabaseFactory {
    public Connection createConnection() { return new OracleConnection(); }
    public Command createCommand() { return new OracleCommand(); }
}

// Client Core Service (Completely decoupled from specific database engines)
class MigrationEngine {
    private Connection connection;
    private Command command;

    public MigrationEngine(DatabaseFactory factory) {
        // Enforces that the connection and command objects always belong to the same family
        this.connection = factory.createConnection();
        this.command = factory.createCommand();
    }

    public void runMigration(String script) {
        connection.establish();
        command.compileAndExecute(script);
    }
}

// Main Driver Demo Class
class AbstractFactoryPatternDemo {
    public static void main(String[] args) {
        System.out.println("--- Bootstrapping Migration Engine for PostgreSQL Pipeline ---");
        DatabaseFactory pgSuite = new PostgresFactory();
        MigrationEngine pgEngine = new MigrationEngine(pgSuite);
        pgEngine.runMigration("CREATE TABLE logs (id SERIAL PRIMARY KEY, note TEXT);");

        System.out.println("\n--- Bootstrapping Migration Engine for Enterprise Oracle Pipeline ---");
        DatabaseFactory oracleSuite = new OracleFactory();
        MigrationEngine oracleEngine = new MigrationEngine(oracleSuite);
        oracleEngine.runMigration("CREATE TABLE logs (id NUMBER GENERATED BY DEFAULT AS IDENTITY, note CLOB);");
    }
}

4. Case 2: Relatable Context in PostgreSQL & Dialect Drivers
If you build enterprise microservices that interact with multiple storage models, or use multi-database migration frameworks like Liquibase or Flyway, the Abstract Factory pattern is the underlying foundation:
- Multi-Dialect Generation Engines: When an ORM (like Hibernate or Sequelize) targets a database engine, it boots a dialect factory module. If the runtime profile points to PostgreSQL, the engine pulls the `PostgreSQLDialectFactory`. This concrete factory guarantees that all auxiliary services—such as JSONB sequence generators, array types handlers, and custom text searches (`ILIKE`)—are generated out of the matching PostgreSQL family. It explicitly blocks the application from accidentally triggering a MySQL or SQL Server syntax module, which would immediately throw runtime exceptions.
- Cross-Platform Analytics Pipelines: Imagine building a dashboard tracking database metrics. An Abstract Factory can deliver matching query and reporting objects designed specifically to pull low-level telemetry logs from native PostgreSQL `pg_stat_activity` views, while cleanly swapping components if the analytics pipeline hooks up to a different data engine.

5. When to Use
- When your system needs to interact with various families of related products, and you want to ensure the application does not depend on or hardcode concrete product implementations.
- When you want to enforce a structural constraint where a suite of objects must be utilized together to maintain application consistency and prevent cross-type bugs.
- When you are designing framework libraries or software development kits (SDKs) where you want to expose object interfaces while keeping their implementation hidden from consumer classes.
