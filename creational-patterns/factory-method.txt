The Factory Method Pattern

1. Explanation
The Factory Method pattern is a creational design pattern that provides an interface for creating objects in a superclass, but allows subclasses to alter the type of objects that will be created. Instead of the client calling `new ConcreteProduct()` directly—which tightly couples code to specific implementations—the client delegates creation to a specialized factory method.

Real-World Analogy: Think of a massive logistics and shipping franchise. A base corporate branch defines a standard workflow: "Receive order, package it, create transport, ship to destination." The corporate office doesn't hardcode how the transport is built because local needs differ. A coastal branch overrides the transport creation method to provision a Cargo Ship, whereas a landlocked branch overrides it to provision a Delivery Truck. The overarching logistics workflow remains identical; only the concrete vehicle shifts based on who is doing the creation.

2. Components Involved
- Product (Interface): Defines the common blueprint for all objects the factory creates (e.g., `DatabaseConnection`).
- Concrete Product: Specific implementations of the product interface tailored to different platforms or configurations (e.g., `PostgresConnection`, `MockDBConnection`).
- Creator (Abstract Class/Interface): Declares the core abstract factory method that returns a Product object. It often contains primary business logic that relies heavily on the Product interface.
- Concrete Creator: Subclasses that override the abstract factory method to manufacture and return a specific variant of the Concrete Product.

3. Code Example: Pluggable Database Environment Connector

// Product Interface
interface DatabaseConnection {
    void connect();
    void execute(String sql);
}

// Concrete Product 1: Production PostgreSQL Connection
class PostgresConnection implements DatabaseConnection {
    @Override
    public void connect() {
        System.out.println("Opening physical secure TCP connection to remote PostgreSQL instance...");
    }

    @Override
    public void execute(String sql) {
        System.out.println("PostgreSQL Engine executing query: " + sql);
    }
}

// Concrete Product 2: Local testing / Mock Database Connection
class MockDBConnection implements DatabaseConnection {
    @Override
    public void connect() {
        System.out.println("Initializing lightweight, in-memory Mock DB layer...");
    }

    @Override
    public void execute(String sql) {
        System.out.println("Mock DB absorbing local test log instruction: " + sql);
    }
}

// Creator Abstract Class
abstract class DatabaseManager {
    // The Factory Method - subclasses will override this to change the object type
    protected abstract DatabaseConnection createConnection();

    // High-level core business logic that depends entirely on the abstraction layer
    public void runQuery(String sqlInstruction) {
        System.out.println("--- Starting Database Transaction Scope ---");
        DatabaseConnection connection = createConnection(); // Loose coupling in action
        connection.connect();
        connection.execute(sqlInstruction);
        System.out.println("--- Transaction Committed Safely ---");
    }
}

// Concrete Creator 1: For Production Environments
class PostgresManager extends DatabaseManager {
    @Override
    protected DatabaseConnection createConnection() {
        // Enforces setup parameters, pool configurations, and environment secrets
        return new PostgresConnection();
    }
}

// Concrete Creator 2: For Local Testing Pipelines
class MockDBManager extends DatabaseManager {
    @Override
    protected DatabaseConnection createConnection() {
        return new MockDBConnection();
    }
}

// Main Driver Demo Class
class FactoryMethodPatternDemo {
    public static void main(String[] args) {
        // 1. Simulate Production runtime targeting PostgreSQL
        System.out.println("=== APPLICATION STARTING IN PRODUCTION MODE ===");
        DatabaseManager prodManager = new PostgresManager();
        prodManager.runQuery("SELECT user_id, email FROM users WHERE active = true;");

        System.out.println();

        // 2. Simulate Local Automated Test Pipeline execution
        System.out.println("=== APPLICATION STARTING IN TEST MODE ===");
        DatabaseManager testManager = new MockDBManager();
        testManager.runQuery("INSERT INTO audit_logs (event) VALUES ('Test Run Execution');");
    }
}

4. Case 2: Relatable Context in PostgreSQL & JDBC Driver Architectures
In your standard development workflow with PostgreSQL, the Factory Method pattern forms the baseline architecture of database drivers:
- JDBC Driver Abstraction: When you compile code using Java's `java.sql` package, you never write explicit code binding directly to PostgreSQL's native engine internal classes. The Java standard API provides `DriverManager.getConnection(url)`. The runtime scans active libraries, locates the PostgreSQL driver JAR file, and triggers its hidden factory hooks. The driver subclasses intercept this setup phase and seamlessly pass back a `PostgresConnection` wrapper disguised behind the universal `Connection` interface.
- Multi-Tenant / Environmental Decoupling: Imagine managing an enterprise application that scales across multi-tenant environments. A factory mechanism safely intercepts runtime profiles, seamlessly serving a standard PostgreSQL connector pool for heavy transactional write loads, or shifting dynamically to a read-replica connection without your business controllers ever knowing the internal structure changed.

5. When to Use
- When you don't know beforehand the exact types and dependencies of the concrete objects your application code needs to interact with.
- When you want to delegate object creation to a subclass hierarchy so that your primary framework components remain completely decoupled from specific product variants.
- When you want to establish an extensible plugin mechanism or give users of your codebase an easy way to inject customized behaviors into internal processing libraries.
